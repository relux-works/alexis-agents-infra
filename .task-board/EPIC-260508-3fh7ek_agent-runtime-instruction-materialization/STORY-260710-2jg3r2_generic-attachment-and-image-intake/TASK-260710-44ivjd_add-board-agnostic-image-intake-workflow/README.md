# Add a board-agnostic image intake workflow

## Description
Extend the canonical relux-agents-infra source with a reusable image-intake and inspection workflow derived from the physical SIM photo bring-up. Agents must be able to resolve explicit local paths or the generic attachments manifest, keep originals read-only, stage and normalize images in project-local scratch, inspect them directly when the runtime supports vision, use a bounded OCR fallback only when needed, redact sensitive identifiers, and emit filename-linked observations with confidence. Integrate the workflow into source-managed instructions, the source skill, helper tooling, setup/install linkage, documentation, and tests as appropriate.

## Scope
Work only in /Users/alexis/src/relux-works/relux-agents-infra. Preserve every pre-existing dirty change. Reuse or narrowly extend agents-attachments rather than creating project-specific scripts. Support explicit paths as well as manifest-backed inputs. Include macOS HEIC normalization through sips and a clearly detected/documented portable fallback such as ImageMagick; fail clearly when a required converter is unavailable. Do not embed task-board commands, board IDs, resources, statuses, or directory assumptions in the reusable contract. Do not implement unbounded OCR/upscale loops or persist raw sensitive SIM identifiers. Do not commit or stage files.

## Acceptance Criteria
A documented CLI/workflow accepts explicit image paths and generic manifest references; stages inputs under caller-controlled or project-local .temp without modifying originals; emits an auditable source-to-staged mapping, preferably machine-readable; handles HEIC normalization on macOS and documents/tests fallback behavior; tells agents to prefer direct vision and limits OCR to a bounded fallback; requires per-file evidence, confidence, uncertainty, and redaction of ICCID/IMSI/QR/key-like secrets; supports child-agent handoff without any board; installed runtime instructions and the relux-agents-infra skill expose the workflow; setup/install linkage and tests cover the helper; existing dirty changes remain intact; relevant tests, diff checks, and source validation pass.
